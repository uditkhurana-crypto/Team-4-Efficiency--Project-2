Section 01
Problem Statement & Goals

CoinDCX manages a large spot trading book across multiple exchanges and custody vaults (Binance, Huobi, KuCoin, CoinDCX HW, Fireblocks, GateIO, etc.). The existing exposure calculation lives in a manually-updated Excel sheet. Risk managers need to answer three questions fast:

📍
Where is our risk concentrated?
Which tokens have the largest long or short exposure in USD terms?
⚠
What requires immediate attention?
Tokens with exposure above $50,000 on either side need an alert surface.
📈
How did exposure build up?
Time-series view per token showing exposure trajectory over time.

Design Constraints

Phase 1 (now): Zero backend infrastructure cost. The dashboard must work by uploading the existing Excel sheet. No data should leave the user's browser.

Phase 2 (future): When a real-time data engineering pipeline is built (Kafka / Flink / Spark / custom), the dashboard must accept a live feed with zero frontend rework — only the data ingestion function changes.

Always: Deployable by a non-engineer on a free tier. Shareable via a URL. Maintainable without a dedicated DevOps team.

Why not React/Next.js from day one?
React adds significant toolchain complexity (webpack, Babel, npm dependencies) for a dashboard that a risk manager needs to deploy in a day. The vanilla-JS architecture is intentionally designed to be migratable: every render function maps directly to a React component. When the team grows or the feature set demands it, the migration is structural — not a rewrite.
Why not Google Sheets / Looker / Tableau?
Off-the-shelf BI tools don't support the custom alert ticker, inline treemap-to-chart click interactions, or the future real-time WebSocket feed. Building custom means full control over interaction model, branding, and data pipeline integration — with no per-seat licensing cost.

Section 05
Dashboard Design Decisions

Upload-First, No Stale Data

The dashboard loads blank. No data is embedded or cached. This was a deliberate choice:

  • Forces the user to always load a fresh snapshot — no risk of acting on stale figures
  • No sensitive financial data committed to version control or visible in the HTML source
  • Clean separation: the app is code, the data is data

Alert Threshold: $50,000

The $50K threshold was set to surface material positions worth attention while filtering noise from small residual balances. This is configurable in the code as a single constant. Future enhancement: user-adjustable threshold per token category.

Simulated Timeseries

The current timeline chart shows a simulated 30-day buildup because the Excel snapshot is a single point in time. In Phase 2, the backend will return historical snapshots from the database and this function will be replaced with a real API call. The chart UI, axis formatting, and interaction pattern will not change.

Treemap + Table Dual View

The treemap gives spatial intuition: which tokens dominate the risk surface at a glance. The table below it gives exact figures for auditing and drilling down. Both are synchronized — selecting in either updates the timeline chart and highlights the selection in the other.

Section 06
Future: Real-Time Pipeline Integration

The dashboard is architected to integrate with any real-time data engineering pipeline with minimal changes. Here is the target architecture:

┌─────────────────────────────────────────────────────────────┐
│                    DATA ENGINEERING LAYER                   │
└─────────────────────────────────────────────────────────────┘

Exchange APIs ──→  Kafka Producer  ──→  Topic: exposure_raw
                       ↓
                  Flink / Spark  ──→  Aggregation, netting,
                       ↓                    position calculation
               TimescaleDB    ──→  exposure_snapshots table
                       ↓
            Express API    ──→  GET /api/exposure/latest
                       ↓                    GET /api/exposure/history?token=BTC
               WebSocket    ──→  ws://api/exposure/stream
                       ↓
        fetchData()    ──→  Same normalized schema
                       ↓
            Dashboard renders (zero change)
📋
Engineering work required for Phase 2: (1) Deploy TimescaleDB on Railway. (2) Write a Python ingestor that reads exchange APIs and writes to the DB. (3) Add 2 Express routes to the existing server.js. (4) Change one line in fetchData() in the dashboard. The entire frontend reuses as-is.
Section 07
Deployment (Railway)

Project Structure

exposure-dashboard/
├── index.html          ← the entire dashboard
├── server.js           ← 15-line Node.js file server
└── package.json        ← { "start": "node server.js" }

server.js

const http = require('http');
const fs   = require('fs');
const path = require('path');

const PORT = process.env.PORT || 3000;

http.createServer((req, res) => {
  fs.readFile(path.join(__dirname, 'index.html'), (err, data) => {
    if (err) { res.writeHead(404); res.end('Not found'); return; }
    res.writeHead(200, { 'Content-Type': 'text/html' });
    res.end(data);
  });
}).listen(PORT, () => console.log(`Running on port ${PORT}`));

Deploy Steps

  1. Push folder to a private GitHub repo
  2. Go to railway.app → New Project → Deploy from GitHub
  3. Select the repo → Railway auto-detects Node.js → deploys
  4. Settings → Domains → Generate Domain → share the URL
Section 08
FAQ — Tech Q&A for Presentations
Is the financial data safe? Where is it stored?
Data never leaves the user's browser. When you upload the Excel file, it is read by JavaScript running locally on your machine. Nothing is transmitted to any server. The server's only job is to serve the HTML file itself — it never receives or processes any exposure data. This is verifiable by inspecting network traffic in browser DevTools.
How do we switch to real-time data without rebuilding the dashboard?
The fetchData() function is the single integration point. In Phase 1 it reads the file. In Phase 2 it calls fetch('/api/exposure/latest'). In Phase 3 it subscribes to a WebSocket. The 2,000 lines of dashboard code — charts, tables, alerts, treemap — remain entirely unchanged. Only the one function that gets data changes.
Why not use a BI tool like Tableau or Looker?
BI tools are excellent for ad-hoc analysis but have fundamental limitations for this use case: (1) They cannot implement the custom alert ticker and click-to-timeline interaction pattern. (2) They have per-seat licensing costs. (3) They cannot be embedded in internal tooling or extended with custom logic. (4) They cannot connect to arbitrary WebSocket streams. A custom dashboard costs engineering time upfront but provides complete control and zero ongoing cost.
What is the total cost to run this?
Phase 1: $0. Railway's free tier provides $5/month in compute credits, which is more than sufficient to serve a static HTML file. Netlify's free tier costs $0. Phase 2 (with TimescaleDB): approximately $15–25/month on Railway (hobby plan for the DB). Phase 3 (with Kafka): $50–200/month depending on message volume and cloud provider.
How does the treemap sizing work?
Each token's rectangle area is proportional to its absolute exposure in USD. The largest exposure gets the most visual space. Tokens are arranged in a 2-row layout for the top 10. Long positions are rendered in shades of green, short positions in shades of red. Clicking any cell selects that token and loads its timeline chart.
Why is the timeline chart simulated?
The Excel sheet is a point-in-time snapshot with no historical data. The simulated timeseries shows what the buildup curve would look like with real historical snapshots. Once the pipeline stores daily/hourly snapshots in a database, the chart function fetches real data from /api/exposure/history?token=BTC&days=30. The chart rendering code does not change — only the data source changes.
Can multiple users view the dashboard simultaneously?
Yes. The server serves a static HTML file with no shared state — each user's session is entirely independent, running in their own browser. There is no database or session management in Phase 1. In Phase 2, the database stores the canonical exposure data and all users see the same figures via the API.
